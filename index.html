02 · Motivation
Why build from scratch?
?
The problem
Libraries hide everything
You call malloc() — but what actually happens? Where does that memory come from? What's the block header structure? Standard libraries answer none of this.
!
The gap
Theory without implementation
OS courses teach paging, scheduling, and file systems — as diagrams. Jarvis-miniOS is where that theory becomes running code you wrote yourself, every single line.
→
The goal
Own the full stack
Build every layer — heap, scheduler, VFS, shell — with zero shortcuts. Understand every decision, every trade-off, every pointer cast. No black boxes allowed.
03 · Architecture
Four-layer system stack
Shell · main.c · ui.c
User-facing interface — tokenize → handle_command → shell_run loop
tokenize() handle_command() ui_dashboard() ui_boot_sequence() ui_set_theme()
System Calls · system.c
Standardised wrappers — intercepts all calls when trace mode is on
sys_alloc / sys_free sys_touch / sys_rm sys_create_process / sys_kill sys_print / sys_ping
Services · memory.c · fs.c · process.c · net.c
Core OS subsystems — each module independent, no cross-calls
mem_alloc / mem_free fs_touch / fs_write / fs_chmod process_schedule() net_ping / net_send
Virtual Hardware · screen.c · keyboard.c · math.c · string.c
HAL — 1MB RAM array, ANSI terminal, non-blocking I/O, custom libs
1MB ram[] array screen ANSI termios non-blocking math.c / string.c
↑ Each layer only communicates with the one directly below it — strict separation of concerns.
04 · Core Features
What's under the hood
🧠
Memory Allocator
Block-table allocator over a 1MB ram[] array. Tracks each block's start index, size, and PID owner. mem_alloc finds the next free slot; mem_free clears the active flag. Inspect live with meminfo.
📂
Virtual File System
In-memory file table mapping names to memory pointers. touch, write, read, chmod, rm, ls — all simulated. fs_write calls sys_alloc to store file data in RAM.
⚙️
Cooperative Scheduler
Round-robin scheduler ticks every 5ms via process_schedule(). Each task is a function pointer — it runs, then yields. Commands: run, kill, top. Background tasks like counter write to VFS concurrently.
🔬
Diagnostics & Trace
trace on intercepts every sys_print / sys_alloc call — like strace in Linux. memtest stress-tests the allocator. meminfo shows live heap block state.
05 · Differentiators
What sets it apart
📦
No Standard Libraries
Zero <string.h>, <stdlib.h>, or malloc. Custom string.c and math.c built from scratch — proves foundational OS understanding.
⚡
Bitwise Math Library
Multiply, divide, power, sqrt — all at O(log N) using bit shifts. Russian Peasant Multiplication, Binary Long Division, Exponentiation by Squaring — no loops.
⌨️
Non-blocking I/O Shell
termios raw mode + POSIX select() lets the shell read keystrokes without blocking — enabling background tasks to run concurrently on the same thread.
📡
Trace Mode (like strace)
Every sys_print and sys_alloc call is intercepted and logged in real time when trace is on — full visibility into the kernel→hardware data pipeline.
📊
Live Dashboard + Themes
One command renders RAM usage, process count, and system health with a progress bar. Three switchable ANSI themes: hacker, retro, ocean — live, no restart.

07 · Challenges & Learnings
What it actually taught me
01
Pointer arithmetic
Working at the byte level
Casting void*, tracking block start indices into the ram[] array by hand, managing alignment with math_align_up — C makes you think in bytes, not objects.
02
Allocator design
Every design choice has a cost
Using a block-table over a 1MB array keeps things simple, but fragmentation is a real problem — free a block in the middle and that gap stays unless a new alloc fits exactly. No coalescing means OOM can happen with free space still available.
03
Non-blocking shell
Cooperative multitasking in pure C
Getting termios raw mode + select() to work so the scheduler could tick every 5ms while the shell stayed responsive — without threads or mutexes — was the trickiest integration problem.
04
Debugging without tools
Build what you need
No gdb in a simulated OS. trace mode wasn't in the original design — I built it because I had no other way to see data moving between system.c and memory.c. Necessity built the feature.
