02 · Motivation
Why build from scratch?
?
The problem
Libraries hide everything
You call malloc() — but what actually happens? Where does that memory come from? What's the block header structure? Standard libraries answer none of this.
!
The gap
Theory without implementation
OS courses teach paging, scheduling, and file systems — as diagrams. Jarvis-miniOS is where that theory becomes running code you wrote yourself, every single line.
→
The goal
Own the full stack
Build every layer — heap, scheduler, VFS, shell — with zero shortcuts. Understand every decision, every trade-off, every pointer cast. No black boxes allowed.

04 · Core Features
What's under the hood
🧠
Memory Allocator
Custom malloc / free with free-list, block splitting, and coalescing. Every allocation tracked. Heap state visible with meminfo.
📂
Virtual File System
Inode-based VFS. touch, write, read, ls — all fully simulated, zero real disk access. Clean file lifecycle from creation to deletion.
⚙️
Process Scheduler
Cooperative multi-tasking with a real process table. fork, exec, kill, ps — processes have PIDs, states, priorities, and runtimes.
🔬
Diagnostics & Trace
Trace mode captures every syscall with timestamps. memtest stress-tests the allocator across thousands of cycles. Built for debugging, not after the fact.
05 · Differentiators
What sets it apart
🖥️
Themed Terminal UI
ANSI color themes, styled boot sequence, and a consistent prompt — it looks and behaves like a real OS shell, not a toy program.
🔍
Transparent Memory Lifecycle
Every block on the heap is visible. Fragmentation, coalescing, free lists — all inspectable live. No black-box allocator.
📡
Trace Mode
Flags every syscall with timestamps. Wasn't planned — I built it because I needed it to debug the scheduler. Now it's a first-class feature.
📊
Live Dashboard
One command. Full OS snapshot: processes, memory usage, open files, uptime. The most impressive view in the entire system.
🌐
Network Simulation
Simulated packet routing and socket model. No real network required — pure software abstraction of networking primitives.

07 · Challenges & Learnings
What it actually taught me
01
Pointer arithmetic
Memory layout in C
Casting void*, computing block headers by hand, managing alignment — C forces you to understand memory layout at the byte level. No abstraction to hide behind.
02
Allocator design
Every decision has a cost
First-fit is fast but fragments. Best-fit reduces waste but costs scan time. Coalescing is critical but hard to get right. Every allocator choice is a performance trade-off.
03
Interface design
Modular systems thinking
The shell shouldn't know how the allocator works internally. Designing clean boundaries between layers — so each component is independently testable — was the hardest non-technical challenge.
04
Debugging without tools
Build what you need
No gdb shortcuts in a simulated OS. Trace mode wasn't planned — I built it because I had no other way to see what was happening. The diagnostics layer exists because I needed it.
